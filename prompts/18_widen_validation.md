## Mission

Push **N(burned assets) across AOIs into the dozens** so lift / precision /
Spearman become interpretable, and so the score can be *calibrated*, not just
*reported*. Do this by adding **4 new, bigger, fire-prone mainland-PT validation
AOIs** and running the existing pipeline on each. **The pilot
(`data/aoi/pilot.geojson`) stays FROZEN** (non-negotiable #10) — these are
*additional* AOIs for validation only.

## Phase A — Author + verify the 4 AOI geojson files

> See `docs/operationalization.md` §5b for the proposed four and the rationale.
> **Three already exist on disk** as alternative-set archives — promote them;
> only the WUI AOI is new. **All bboxes are DRAFT until verified** (#1).

The four (slugs → new `data/aoi/<slug>.geojson`):

1. `pedrogao_grande` — promote `data/aoi/alt_pedrogao_grande.geojson` (2017 mega-fire, large scar).
2. `serra_da_estrela` — promote `data/aoi/alt_serra_da_estrela.geojson` (2022, EMSR618).
3. `peneda_geres` — promote `data/aoi/alt_peneda_geres.geojson` (2025, Norte-mountain shift; **preliminary** ICNF figures — finalise).
4. `monchique` — **author new**, WUI axis (2018, EMSR303). Tighten bbox around the scar before freezing.

For **each** AOI, the verification gate (all must pass before the geojson is
declared final):

- **(a) Audit GREEN:** `uv run wildfire-exposure-eo audit` (or `audit-all`) on the
  AOI → S2/S1/Cop-DEM/ETH-GCH/OSM/ICNF/WorldCover all reachable. Commit the audit
  JSON sample under `docs/samples/`.
- **(b) ICNF intersection:** post-window ICNF perimeters **actually intersect**
  the bbox (else the AOI adds assets but zero burn labels — useless). Log the
  perimeter count + hectares inside the AOI.
- **(c) Figure provenance:** any burned-area number cites **ICNF or EFFIS
  explicitly** — never average the two (they use different methodologies).
- **(d) OSM density check:** non-trivial critical-infra count (high-mountain AOIs
  are thinner — verify before relying on them).

Each AOI ships a matching `smoke_<slug>.geojson` (three already exist; author one
for `monchique`). Keep CRS explicit (CRS84) in every geojson.

## Phase B — Run the pipeline + combined validation

For each AOI, run the full path (`fetch-osm → fetch-rasters → fetch-burns →
fuel-layer → score → validate`) with the **same** leakage-clean harness
(`validation.assert_no_temporal_leakage`, backdated window, strictly-post-window
ICNF vintage). **Recommended: run with the pillar-0 seasonal feature and pillar-1
topology features active** if they have landed — that is the run worth reporting.

Deliverables:

- New `scripts/18_validate_multi_aoi.py` — runs validation per AOI and emits a
  **combined report** that aggregates N(burned assets) across AOIs, with per-AOI
  and pooled lift / Spearman. House rule: every number script-generated, with the
  `<!-- generated by: scripts/18_validate_multi_aoi.py at <commit> -->` citation.
- `docs/validation_report.md` extended (or a new `docs/validation_multi_aoi.md`,
  cross-linked) with the pooled table. State **honestly** whether N is now high
  enough that the ablation resolves which features carry the signal — if it still
  doesn't, say so.
- AOI-relative caveat: scores are per-AOI percentile ranks, so pooling for lift
  must be done carefully (pool the binary burned-label vs the within-AOI rank, or
  document the chosen pooling). Make the method explicit; do not silently mix
  AOI-relative ranks as if globally comparable.
